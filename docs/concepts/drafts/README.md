## Назначение папки

Здесь хранятся идеи и черновики, которые требуют анализа перед интеграцией в SSOT или другую документацию проекта.

**Правило**: Ничего из этой папки не является утверждённым. Это материал для обсуждения.

---

## Содержимое

### 1. Оркестратор — Когнитивный стек

**Файл**: [ORCHESTRATOR_COGNITIVE_STACK_v0.1.md](ORCHESTRATOR_COGNITIVE_STACK_v0.1.md)

**Ключевые идеи**:
- State-Machine для задач (PENDING → RESEARCHING → EXECUTING → CRITIQUE → FINALIZED)
- Internal Economy (Resource Budget) — лимиты по итерациям и деньгам
- SSOT как "живой организм" — машиночитаемый граф
- "Triangle of Quality" — Исполнитель + Критик + Арбитр

---

### 2. Оркестратор — Динамика мышления

**Файл**: [ORCHESTRATOR_BRAIN_DYNAMICS_v0.1.md](ORCHESTRATOR_BRAIN_DYNAMICS_v0.1.md)

**Ключевые идеи**:
- Три режима мышления: Top-Down декомпозиция, Quality Loop, Self-Reflection
- Integrity Guard — оркестратор как хранитель целостности
- Post-mortem отчёты и причинно-следственные связи в логах
- Боевой сценарий для валидации архитектуры

---

### 3. Концепция Оркестратора v0.2 (полный черновик)

**Файл**: [ORCHESTRATOR_CONCEPT_v0.2.md](ORCHESTRATOR_CONCEPT_v0.2.md)

**Ключевые идеи**:
- "Процессор и ISA" — метафора оркестратора как CPU
- Три режима Human-in-the-loop (A/B/C)
- Полные этапы работы: Инициация → Правила → Концепция → Аналитика → Документация → План → Подбор команды → Исполнение → Логи
- Интеграция с рецензией MESSAGE_FORMAT
- Сложности и пути решения (8 пунктов)

---

### 4. Интерпретатор Process Card

**Файл**: [PROCESS_CARD_INTERPRETER_v0.2.md](PROCESS_CARD_INTERPRETER_v0.2.md)

**Ключевые идеи**:
- Движок оркестратора = парсер YAML + DSL-семантика
- Компиляция в IR/AST (граф шагов)
- Семантика шагов: action, condition
- Переменные и контролируемое состояние
- Безопасность: safe-load YAML, ограниченные выражения
- Тестирование "процессора" (Unit/Integration/E2E)

---

### 5. Человеческие имена агентов

**Файл**: [AGENT_HUMAN_NAMES_v0.1.md](AGENT_HUMAN_NAMES_v0.1.md)

**Ключевые идеи**:
- Два слоя идентификации: системный (UID) и человеческий (display name)
- "Перекличка" агентов для человека
- Три варианта решения (A/B/C) с минимальными изменениями в SSOT
- Display Identity как функция монитора, не оркестратора
- Коллизии имён и их обработка

---

## Что делать с этими черновиками

1. **Проанализировать** каждый документ на соответствие текущей архитектуре
2. **Выделить** идеи, которые уже реализованы в SSOT
3. **Выделить** идеи, которые стоит добавить в SSOT или Roadmap
4. **Отклонить** идеи, которые противоречат принципам проекта
5. **Зафиксировать** решения в соответствующих документах
